# Project BirdMan (PBM) — 最上位仕様書

本書はPBMの最上位仕様である。下位文書(REQUIREMENTS.md ほか)と矛盾する場合、本書を優先する。
出典: PBM Mission (2026-07-01) / PBM Vision (2026-07-02) / チーム合意の共通プロジェクト仕様。

## 1. 目的

PBMは、鳥人間コンテストに出場するチームが人力飛行機の設計で次の状態に陥ることを防ぐ。

- 何から設計を始めればよいか分からない
- どの解析ソフトを使用すればよいか分からない
- 計算方法や過去の設計根拠が継承されていない
- XFLR5、XROTOR、表計算、CADが分断されている
- 試験飛行の機会が少なく、設計の妥当性を確認できない
- 設計変更による影響を追跡できない

3本柱: **Design**(設計を支援する)、**Knowledge**(知識を継承する)、**Simulation/Challenge**(挑戦を後押しする)。

Mission: チームがデータ管理・継承管理ではなく「より良い機体を設計すること」に集中できる環境を提供する。設計思想・変更理由・解析結果・試行と改善の記録を資産として残す。まず自チームで実運用し、その実績をもとに全国の鳥人間チームへ展開する。

## 2. 重要な設計方針(人間の承認)

PBMは「AIが勝手に完成機を決定するアプリ」ではない。以下の工程には必ず人間の承認を設ける。

1. 設計要求の確定
2. 使用材料と製造方法の確定
3. 荷重条件の確定
4. 安全率の確定
5. 最適化結果の採用
6. 最終形状の承認
7. 製造図面の承認
8. 飛行試験の実施判断

計算結果には必ず、使用した式・単位・入力値・仮定・出典・警告・適用範囲を表示する。
**解析を実行していない場合は、実行したように見せてはいけない。**

## 3. 想定ユーザー

大学の鳥人間チーム。特に、歴史が浅い/設計経験者が少ない/引き継ぎ資料が不足している/XFLR5・XROTORの使用経験が少ない/飛行試験を十分に行えないチーム。

## 4. 設計工程(Step 1〜13)

| Step | 内容 | 要点 |
|---|---|---|
| 1 | プロジェクト作成 | チーム名・機体名・設計年度・部門・設計責任者・単位系・バージョン・設計目標 |
| 2 | 要求仕様入力 | パイロット質量/出力、目標速度/距離、翼幅制限、質量目標、離陸/旋回/風速条件、製造制約、材料、安全率、大会規則。**物理量は値と単位を一体で管理** |
| 3 | 初期サイジング | 全備質量、必要揚力、翼面積、翼面荷重、AR、失速速度、必要推力/動力、誘導/有害抗力、L/D、Re、推進効率、出力収支。初期値・推奨範囲・制約違反を表示 |
| 4 | 主翼設計 | 翼型、翼幅、翼弦、テーパー、後退角、上反角、ねじり下げ、桁位置等をパラメトリック管理 |
| 5 | 空力解析 | XFLR5アダプター。入力生成→実行→結果取得→内部形式変換→グラフ/表→設計案比較。**モックと実解析を明確区別** |
| 6 | プロペラ設計 | XROTORアダプター。回転数・直径・枚数・目標推力等 → 推力・トルク・効率・荷重分布・設計点/非設計点性能 |
| 7 | 安定性・操縦性 | 重心、空力中心、静安定余裕、尾翼容積、操舵面。安定性NG案は最適化候補から除外可能 |
| 8 | 構造設計 | MVPは簡易梁モデル。曲げ/せん断/たわみ/座屈/ねじり/接合部/安全率。式・境界条件・荷重条件・単位を必ず表示。将来FEM連携 |
| 9 | 質量・重心管理 | 部品単位(名称、カテゴリ、質量、座標、材料、推定/実測、不確かさ、担当、更新日時)。総質量・重心・慣性モーメント・目標差・重心移動 |
| 10 | 統合性能計算 | 空力+推進+構造+出力+安定性 → 飛行速度、滑空性能、離陸可能性、航続距離、エネルギー収支 |
| 11 | 設計最適化 | 設計変数・制約・評価関数を分離。パレート解の比較 |
| 12 | CAD連携 | 段階的: パラメータ出力→CSV/JSON→DXF/STEP/STL→Fusion 360スクリプト→API直接連携。CAD失敗で設計計算全体を停止させない |
| 13 | 設計レポート | 概要、要求、入力、仮定、式、解析結果、質量重心、安定性、案比較、制約違反、未検証事項、変更履歴、要承認事項。PDF/HTML/CSV |

## 5. Knowledge機能

設計判断、採用/不採用理由、参考文献、過去機体比較、失敗事例、製造上の注意、試験結果、変更履歴、判断者、判断日時を年度をまたいで管理する。AIは過去設計を検索・提案できるが、過去の判断を絶対的な正解として扱わない。

## 6. Simulation機能(将来拡張)

飛行軌道、離陸、旋回、突風応答、パイロット出力変化、操舵入力、質量変化、モンテカルロ、不確かさ解析、フライトシミュレーター連携。

## 7. 技術構成

- フロントエンド: Next.js + TypeScript + React
- バックエンド: Python + FastAPI + Pydantic + NumPy + SciPy + Pandas
- DB: SQLite(開発初期)→ PostgreSQL(将来)
- テスト: pytest(単体・統合・数値回帰)、Playwright(主要画面)
- 開発環境: Docker / Docker Compose / GitHub Actions / 環境変数設定 / Windows動作可 / 外部解析ソフトパス設定可

## 8. システム構造

計算ロジックをUI・外部ソフトから分離する。モジュール: domain / calculation / workflow / optimization / adapters / api / frontend / persistence / reports / tests / docs。外部ソフトは必ずアダプター経由。

## 9. 単位管理

- SI単位系を内部標準とする。入力時に単位選択可。値と単位を一体で保持
- API境界で単位を検証。計算途中の暗黙変換禁止。グラフ・レポートに単位明記
- 次元不一致の計算を検出。Pint等の単位ライブラリを使用

## 10. AI機能の制限

AIができること: 設計案提案、入力不足指摘、異常値検出、過去設計検索、案比較、レポート文章作成、次の解析提案。
AIに直接決定させないこと: 未検証の材料強度、根拠のない安全率、存在しない解析結果、大会規則適合判定、最終飛行安全判定、最終製造承認。
AI出力は計算エンジンの結果と分離して保存する。

## 11. 品質要件

計算式にテストがある / 代表的な手計算と一致する / 入出力を追跡できる / エラーを握りつぶさない / 外部ソフト失敗時ログ保存 / モックと実解析の明確区別 / 有効桁と単位 / 変更前後比較 / 破壊的変更を勝手に実行しない / 秘密情報をリポジトリに保存しない。

## 12. 開発段階

- **Phase 0** 基盤設計: 要求定義、ドメインモデル、データモデル、アーキテクチャ、開発/テスト環境、サンプルデータ
- **Phase 1** 初期サイジングMVP: プロジェクト作成、要求入力、単位管理、初期サイジング、簡易空力計算、質量管理、HTMLレポート
- **Phase 2** 解析ソフト連携: XFLR5/XROTORアダプター、ジョブ管理、結果比較、エラー処理
- **Phase 3** 構造・安定性: 簡易梁、重心、静安定余裕、尾翼サイジング、制約判定
- **Phase 4** 最適化: 設計変数管理、制約付き最適化、パレート解、感度分析、不確かさ分析
- **Phase 5** CAD・シミュレーション: CADパラメータ出力、Fusion 360スクリプト、飛行シミュレーション、レポート拡張

## 13. 開発ルール

一度に全機能を実装しない / 各Phaseを動作可能な状態で完成 / 大きな変更は小さく分割 / 変更前に既存コード確認 / 既存テストを壊さない / 計算式をUIに書かない / 係数を根拠なくハードコードしない / 不明値は仮定として明示 / TODO大量生成で完了扱いしない / 動作未確認を完成報告しない / Windowsパス考慮 / 外部ソフトなしでも基本機能テスト可。

## 15. 最終目標

要求仕様入力 → 初期設計案生成 → 空力/推進/構造/安定性計算 → XFLR5/XROTOR実行 → 制約違反検出 → 複数案最適化 → 人間へ提示 → 承認案からCADデータ生成 → 設計根拠レポート化 → 次年度への知識継承。
**ただし、未検証の計算結果を用いた実機製造や飛行を自動承認してはならない。**
